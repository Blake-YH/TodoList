# 轻量化 PC TodoList 软件规划文档

## 1. 项目目标

开发一款面向个人使用的轻量化 PC TodoList 软件，核心目标如下：

- 快速打开，低内存占用，低打扰
- 以“记录待办并迅速完成”为核心，不做复杂项目管理
- 支持离线使用，基础数据本地可靠保存
- 后续可以逐步扩展提醒、同步、统计等功能，但首版不为扩展性过度设计

本项目首先应追求的是：

1. 易开发
2. 易维护
3. 安装包和运行成本可接受
4. 用户体验足够轻快

不是首先追求：

- 企业级协作
- 复杂权限系统
- 多端实时同步
- 过重的本地数据库和后端体系

---

## 2. 产品定位

### 2.1 目标用户

- 个人用户
- 需要在 Windows PC 上管理日常待办的人
- 希望比网页更方便、比纸笔更结构化、比复杂生产力工具更轻的人

### 2.2 核心使用场景

- 快速新增待办
- 查看“今天要做什么”
- 标记完成 / 取消完成
- 调整优先级
- 通过分类或标签做简单整理
- 下次打开时，数据仍然完整存在

### 2.3 产品边界

首版建议只做单机版，不做账户系统与云端同步。

原因：

- 轻量化产品的价值首先来自“快”和“稳”
- 账户、同步、冲突解决、服务端部署会显著提高复杂度
- 在需求尚未验证前，这类投入性价比低

---

## 3. 核心需求分析

## 3.1 功能需求（首版必须具备）

### Todo 基础能力

- 新建待办
- 编辑待办标题与描述
- 删除待办
- 标记完成 / 未完成
- 设置截止日期
- 设置优先级：低 / 中 / 高
- 支持分类或标签（二选一即可，首版更建议分类）

### 列表与筛选

- 查看全部待办
- 查看未完成待办
- 查看已完成待办
- 查看今天到期 / 逾期待办
- 按优先级、分类筛选

### 数据存储

- 本地持久化保存
- 软件重启后数据仍然保留
- 至少具备基础的数据导出能力

### 桌面端体验

- 支持窗口化使用
- 启动速度快
- 基础键盘操作顺畅

## 3.2 第二阶段需求（建议预留，但不进入 MVP）

- 系统托盘
- 开机自启动
- 到期提醒 / 通知
- 搜索
- 快捷键快速新增
- 深色模式
- 数据导入 / 备份恢复

## 3.3 暂不建议纳入首版

- 多人协作
- 云同步
- Markdown 富文本编辑
- 看板 / 甘特图 / 日历全景视图
- 插件系统

原因是这些需求会直接把产品从“轻量 Todo 工具”推向“综合效率平台”，与项目目标冲突。

---

## 4. 非功能需求

### 4.1 性能目标

- 冷启动尽量控制在 2 秒内
- 常驻内存尽量控制在 150MB 以内
- 日常数据规模（1000 条以内 Todo）下交互无明显卡顿

说明：

对于桌面应用来说，绝对极致轻量通常意味着使用原生 GUI；但原生 GUI 会显著提高开发门槛与 UI 开发成本。因此这里要追求的是“足够轻量”，而不是“极限轻量”。

### 4.2 可维护性

- 前后逻辑尽量简单
- 技术栈稳定，社区成熟
- 便于后续加入通知、托盘、自动更新等桌面能力

### 4.3 可靠性

- 本地数据不易损坏
- 关闭应用后数据可恢复
- 异常情况下尽量避免用户数据丢失

### 4.4 可发布性

- 能产出 Windows 安装包
- 后续可支持自动更新

## 5. 技术路线候选与严格选型

要做的是“轻量化 PC 软件”，本质上有 4 条主路线：

1. Electron
2. Tauri
3. .NET 原生桌面（WPF / WinUI）
4. Qt / Flutter 等跨平台原生方案

下面逐项推导。

## 5.1 Electron

### 优点

- 生态最成熟
- 桌面能力完善
- 前端开发体验最好
- 文档与社区资源丰富

### 缺点

- 安装包偏大
- 内存占用通常高于 Tauri 和原生方案
- 对“轻量化”目标不够友好

### 结论

如果目标是最快做出产品原型，Electron 是非常稳妥的选择。
但如果“轻量化”是明确核心目标，Electron 不是最优解。

## 5.2 Tauri

### 优点

- 安装包和内存占用通常优于 Electron
- 前端仍可使用熟悉的 Web 技术栈
- 原生能力通过 Rust 提供，性能和安全性较好
- 很适合做工具类桌面应用

### 缺点

- 需要同时接触前端 + Rust / Tauri 桌面层
- 某些生态成熟度不如 Electron
- 团队若完全没有 Rust 基础，前期认知成本略高

### 结论

如果目标是在“开发效率”和“轻量化”之间取得较好平衡，Tauri 是当前最合适的候选。

## 5.3 .NET 原生桌面（WPF / WinUI）

### 优点

- Windows 桌面集成度高
- 运行效率和系统适配通常较好
- 更像传统原生桌面软件

### 缺点

- UI 开发体验对前端开发者不够友好
- 跨平台能力弱
- 社区内容相对更偏 Windows 桌面老体系

### 结论

如果项目明确只做 Windows，且团队熟悉 C# / XAML，这是可行方案。
但如果从 0 到 1 追求效率，并希望未来保留跨平台可能性，不是当前最优解。

## 5.4 Qt / Flutter

### 优点

- 可以做出较接近原生体验的跨平台桌面应用
- 性能通常不错

### 缺点

- 技术栈与常规 Web 前端差异较大
- 学习与工程搭建成本高于 Tauri + Web
- 对“先验证产品”来说偏重

### 结论

适合有对应技术积累的团队，不适合当前这个以效率和轻量为目标的首版项目。

## 6. 最终推荐技术方案

## 6.1 总体结论

推荐采用：

- 桌面容器：Tauri
- 前端框架：React + TypeScript + Vite
- UI 方案：原生 CSS / Tailwind CSS（二选一，推荐 Tailwind CSS）
- 数据存储：SQLite
- 状态管理：Zustand
- 表单与校验：React Hook Form + Zod
- 数据访问：Tauri Command + 本地服务层封装
- 打包发布：Tauri Bundle

## 6.2 为什么是这套方案

### 选择 Tauri，而不是 Electron

因为你的目标里“轻量化 PC 软件”是关键约束。Tauri 在包体和资源占用上通常更符合这一要求，同时保留 Web 技术栈的开发效率。

### 选择 React，而不是 Vue / Svelte

React 不是唯一可行方案，但它的优势是：

- 生态最成熟
- 配套库最全
- 后续接入复杂交互、表单、状态管理、测试都更稳

如果你个人非常熟悉 Vue，也可以改成 Vue 3 + TypeScript；但在默认推荐上，React 的通用性更强。

### 选择 TypeScript，而不是纯 JavaScript

Todo 软件虽然业务简单，但会涉及：

- Todo 数据模型
- 筛选状态
- 存储结构
- 桌面端命令调用

TypeScript 可以降低后续迭代时的隐性错误，首版多花一点约束成本，长期更省维护成本。

### 选择 SQLite，而不是 JSON 文件

JSON 文件的优点是简单，但当需求逐渐加入：

- 筛选
- 排序
- 分类
- 截止日期
- 完成状态统计

SQLite 会明显更稳。

它的优势：

- 单文件数据库，仍然是本地轻量方案
- 数据结构清晰
- 查询和扩展能力更强
- 便于未来增加搜索、统计、归档

结论是：SQLite 比 JSON 更适合作为可持续演进的本地存储方案。

### 选择 Zustand，而不是 Redux

该项目状态规模不大，Redux 会引入不必要的样板代码。Zustand 足够轻、足够直观，更符合“轻量工具”项目的工程目标。

## 7. 推荐系统架构

建议采用以下分层：

### 7.1 表现层（Frontend UI）

负责：

- 页面展示
- 表单输入
- 列表筛选
- 交互反馈

推荐页面结构：

- 今日待办页
- 全部待办页
- 已完成页
- 设置页

### 7.2 应用层（Frontend Store / Service）

负责：

- 状态管理
- Todo 的增删改查逻辑编排
- 调用 Tauri command
- 处理前端校验和错误提示

### 7.3 桌面能力层（Tauri）

负责：

- 文件系统访问
- 系统通知
- 托盘
- 自动更新
- 调用本地数据库逻辑

### 7.4 数据层（SQLite）

负责：

- Todo 数据持久化
- 分类数据
- 配置项数据

## 8. 数据模型建议

首版建议使用以下核心表结构。

### 8.1 todos

字段建议：

- id
- title
- description
- status
- priority
- category_id
- due_date
- created_at
- updated_at
- completed_at

说明：

- `status`：`pending` / `completed`
- `priority`：`low` / `medium` / `high`

### 8.2 categories

字段建议：

- id
- name
- color
- created_at

### 8.3 settings

字段建议：

- id
- key
- value

这样设计的好处是：

- 首版足够简单
- 后续可平滑扩展提醒、主题、自启动等配置

## 9. 页面与交互建议

### 9.1 MVP 页面

- 左侧导航：今天、全部、已完成、设置
- 右侧主区域：待办列表
- 顶部区域：快速新增、筛选、搜索入口（搜索可以先占位）

### 9.2 核心交互原则

- 新增 Todo 的路径尽量短
- 完成 Todo 的动作尽量直接
- 不强制用户填写复杂字段
- 非关键字段默认折叠，避免输入负担

### 9.3 轻量化体验的关键

- 默认展示最重要的信息：标题、截止日期、优先级
- 描述信息弱化展示
- 避免一个 Todo 打开独立复杂详情页

## 11. 风险分析

### 11.1 Tauri 学习成本

风险：

- 需要理解前端与 Rust/Tauri 的边界

控制措施：

- 首版尽量让 Rust 端只承担桌面能力与数据库桥接
- 业务逻辑尽量放在前端服务层，减少 Rust 复杂度

### 11.2 本地数据一致性

风险：

- 如果存储方案设计随意，后续会难以扩展

控制措施：

- 一开始就使用 SQLite
- 保持清晰的数据表结构和迁移机制

### 11.3 功能蔓延

风险：

- Todo 工具很容易不断添加日历、番茄钟、笔记、协作等需求

控制措施：

- 坚持 MVP 边界
- 只有在核心待办体验被验证后再扩展

---

## 12. 最终建议

如果你的目标是做一个：

- 真正可用
- 足够轻量
- 开发效率合理
- 后续还能继续迭代

那么当前最推荐的技术方案是：

**Tauri + React + TypeScript + Vite + SQLite + Zustand**

这是在以下几个维度上综合最优的方案：

- 比 Electron 更轻
- 比原生桌面方案更容易上手
- 比 JSON 存储更稳健
- 比重型状态管理更简洁

---

## 13. 立项结论

本项目建议定义为：

> 一个面向个人用户的本地优先、轻量化、以快速记录和完成待办为核心体验的 Windows PC Todo 桌面软件。

首版实现原则：

1. 先单机，后同步
2. 先稳定，后复杂
3. 先 Todo，后效率平台

基于以上推导，正式推荐技术栈为：

**Tauri + React + TypeScript + Vite + SQLite + Zustand**
